# アプリ設計書

## 概要

メイン画面から各機能ページに遷移し、その画面が持つ機能を実行するWebアプリ

## 使用技術

- フロントエンド
  - TypeScript
  - Vite
  - React
  - HTML
  - css

- バックエンド
  - Node.js
  - Express

## 画面一覧

### メイン画面

- 時刻を表示する
- ログイン画面へ遷移する

### ログイン画面

- ユーザ名・パスワードを入力できる
- ログインボタンを押下することで、ダッシュボード画面へ遷移する

### ダッシュボード画面

- 各機能のページへ遷移するボタンを配置する
- 画面右上にユーザ名とIDを記載する

## 構成の説明

- フロントエンド
  └ features … 機能単位でソースコードを管理するディレクトリ。
  | └機能 … 機能ごとにフォルダを作成する。
  |   └ components … UIコンポーネントを配置する。
  |   └ hooks … componentsの状態管理や画面ロジックを配置する。
  |   └ services … APIをなどの外部アクセス処理を配置する。
  |   └ types … 型定義を行う。
  └ pages … 画面単位のコンポーネントを配置する。
  └ services … アプリ全体で使用する共通サービスを配置する。
  └ App.tsx … アプリケーションのルートコンポーネント。
  └ main.tsx … Reactのエントリーポイント。
  └ style.css … アプリ全体の共通スタイルの管理。

- バックエンド
  src … バックエンドアプリケーション本体を配置する。
    └ 機能 … 機能ごとにフォルダを作成する。
      └ controller.ts … HTTPリクエストの受付窓口。
      └ service.ts … ビジネスロジックを管理する。
      └ repository.ts … データアクセスを管理する。
      └ model.ts … データ構造の定義。

  ## 依存関係
  - 依存関係は必ず一方向。
  - フロントエンド
  ```
  Page
  ↓
  components
  ↓
  hooks
  ↓
  services
  ↓
  Backend API
  ```


  - バックエンド
  ```
  Controller
  ↓
  Service
  ↓
  Repository
  ↓
  Model
  ```
  
## テスト

### フロントエンド

- Vitest
- React Testing Library

用途

- Componentの単体テスト
- Hookのテスト
- 画面表示の確認
- ボタン押下時の動作確認

### バックエンド

- Vitest

用途

- Serviceの単体テスト
- Repositoryの単体テスト
- ビジネスロジックの検証

### APIテスト

- Postman

用途

- APIの動作確認
- リクエスト・レスポンス確認

### テスト方針

- 機能単位でテストコードを作成する。
- 実装完了後に単体テストを作成する。
- リファクタリング時に既存テストが成功することを確認する。
